### Changed — Portal separation + onboarding/twin persistence fixes (2026-07-16)
- **Employer and candidate portals are now cleanly separated shells.** Employer routes moved from the candidate group `app/(app)/employer/*` into their own route group `app/(employer)/employer/*` (URLs unchanged) with a dedicated `EmployerShell` (employer-only nav: Dashboard, Post a role, Credits & billing) and a minimal employer topbar. The candidate `Sidebar` no longer bleeds the "For recruiters → Employer Portal" block. Each shell cross-links to the other (candidate profile menu → "Employer portal"; employer sidebar → "Switch to job search"). Signed-out access to `/employer/*` now redirects to `/employer/login` (via `ClientRedirectToLogin` `loginPath` prop) instead of the candidate login.
- **Both-roles decision (was an open question): NOT gated.** One account may hold an independent candidate Career Twin and an employer profile — separate backend namespaces keyed by the same `X-User-Id`. Portals are separated by URL + shell, not by an account role flag.
- **Fix: "recreate my Career Twin on every login."** The dashboard's catch-all redirected to `/create-career-twin` on *any* error from the secondary matches/applications/invites fetches (e.g. a transient `/matches` 502), throwing returning users with an activated twin back into the onboarding wizard. Reworked `dashboard/page.tsx` so the twin fetch + onboarding gate is evaluated first and in isolation — **only a genuinely missing or not-yet-activated twin redirects to onboarding**; a transient backend error falls back to the sample dashboard, and secondary fetches degrade to empty state (`.catch`) and never gate onboarding. `redirect()` is now called after the try so its internal throw is never swallowed. Same antipattern fixed in `career-twin/page.tsx` (transient errors → soft "unavailable" placeholder, not an onboarding bounce).
- **Killed the dead "I'll do this later" skip** on the Career Twin wizard's welcome step. Every candidate page gates on an activated twin, so the skip only bounced the user straight back to the wizard — onboarding is the required first step.
- **Generation now produces real artifacts (PDF/Word), not advice.** The paid web app could only copy draft text to the clipboard, while the free Streamlit v1 (`app.py:182`) had shipped **⬇️ PDF / ⬇️ Word** buttons all along — a straight product regression into the tier people pay for. `core.generate_cv` + `skills/cv_template.md` (a complete, relevance-weighted, ground-truth-only CV) and `core.make_pdf` / `make_docx` already existed and were simply never wired into the Next.js app. Now they are:
  - **New `POST /applications/cv`** — generates a COMPLETE tailored CV via `core.generate_cv`, distinct from the draft's "CV Bullet Points" (fragments, not a CV). One model call; counts against the same monthly allowance (disclosed in the UI per the API-calls guardrail); async job reusing the existing `_generation_jobs` store and `GET /applications/generate/{job_id}` poll endpoint.
  - **New `POST /applications/export`** — renders any section to a real `.pdf`/`.docx` via the existing renderers. No model call and **nothing persisted** — pure rendering of text the caller already holds, so no artifacts-tab persistence was needed to ship downloads. Guards: format allow-list, 256 KB ceiling, and `_export_slug` so user text can never steer the `Content-Disposition` header.
  - **New `core.split_application`** — defensive splitter (house style: never raises; unknown headers degrade to "it's all the cover letter" rather than losing the draft). Mirrored client-side as `splitDraft`.
  - **Fixes a real footgun:** `generated.result` is one blob containing the Fit Evaluation — the candidate's own gap analysis and `(stretch — verify)` markers. Exporting it whole hands users a file that, if forwarded, sends a hiring manager their own weaknesses. The evaluation is now **screen-only and never exportable** (labelled "For your eyes only"); only the cover letter and CV are downloadable.
  - **Correction:** an earlier note here claimed `app.py:182` had the same footgun. It does not — `core.strip_evaluation` has always existed and `app.py` strips before every download (`app.py:214`, `:224`). That claim was wrong and is withdrawn.

### Security — employers could wear a company's "Verified" badge without proving they work there (2026-07-17)
- **The hole.** `verify.verify_employer` scores a **domain**; nothing ever proved the signup had any relationship to it. Demonstrated end-to-end on the real scorer: sign in with any Gmail → onboard as "Paystack" / `paystack.com` → score **90** → level **`high`** → candidates saw **"✅ Verified employer"** (`api/main.py:1604` computes `verified = warm_intro_by or trust_level == "high"`). The first role is free with 10 invites, and the never-pay-a-fee warning only fires on `low` trust — so an impersonator's invite carried **no warning at all**. Our own trust badge was the lure for the scam this product exists to prevent.
- **Root cause is documented in the code** (`api/main.py:1135`): the spec's original free-mail trust cap was dropped *because* Google-only auth means every employer signs up with a personal Gmail — removing the only signal tying a human to a company. The gap is **proof of affiliation, not credentials**; passwords would not have closed it (see `EMPLOYER_DOMAIN_VERIFICATION.md` §1).
- **Stopgap:** `verify.employer_portal_level` gains `domain_verified=False` — **`high` is now unreachable without proven domain control**, so cold signups cap at `medium`. The default is safe, so every caller (onboarding, the refresh worker) is capped without changing a line. Trust *scores* are still stored truthfully; only the claim of verification is withheld. Admin vouch (`warm_intro_by`) is untouched — it remains the one path that legitimately earns the badge, and a test now proves vouching still produces `verified: true`.
- **Backfill:** rows already stored as `high` would have kept the false badge for up to **7 days** (`workers/verify_employers` skips domains whose trust record is fresh, `max_age_days=7`), so `db._migrate` downgrades pre-existing cold `high` rows. It is **self-disabling** — a no-op the moment the `domain_verified` column lands — so it can never re-downgrade a genuinely verified employer. Probes with a read before taking a write lock (workers and the API share one SQLite file on the Render Disk). Vouched employers are never touched.
- Employer dashboard now explains why a legitimate employer isn't verified yet, instead of showing a bare middling badge.
- Tests: +9 (`test_verify` domain-control cap incl. an end-to-end impersonation regression through the real scorer, and that proven control never rescues `avoid`/`low`/red-flags; `test_employer` cold onboarding is never `high`; `test_invites` cold employer never shows a verified badge **and** vouch still does; `test_db` backfill downgrades cold `high`, spares vouches, self-disables). `test_notify_worker`'s digest assertion updated — the honest line now reads `trust medium`.
- **Follow-up specced, not built:** `EMPLOYER_DOMAIN_VERIFICATION.md` — prove domain control via a code to `you@company.com` (Brevo, already wired), with a DNS-TXT fallback. Open decisions for Joy in §6: how hard verification gates invites, whether `medium` invites carry the scam warning, and whether vouched employers skip it.

### Fixed — evaluation-in-the-file footgun, closed everywhere (2026-07-16)
- **The leak was in the new splitter, not the Streamlit app.** `build_prompt` asks for `## Fit Evaluation`, but real output also uses `## Fit Score` (the canned fixture in `test_e2e` proves the variant). `split_application` matched only the first spelling, so a `## Fit Score` section fell through into `cv_bullets` — and got **exported into the CV**. `core` now has ONE definition of an evaluation header (`_EVAL_HEADER`) shared by `split_application` and `strip_evaluation`, so the two can never disagree again. Regression tests cover both variants and assert the sections agree.
- **`POST /applications/export` now strips the evaluation server-side** rather than trusting the caller — a comment saying "don't send the evaluation" is not a guard. Any caller (web, a future client, a direct API user) gets an evaluation-free document; a body that is *only* an evaluation now 422s as "nothing to export" instead of rendering. Verified by rendering a full blob to PDF and extracting the text back: no `Fit Score`, no `Biggest gaps`, sendable content intact.
- **Killed the dead employer email/password form** on `/employer/login`. The submit button was `disabled` and "Sign up with your work email" was a `<span>`, so employers typed real (and often reused) passwords into a form that went nowhere. Google is the product's auth method (`auth.ts`); the page now says so honestly — work email signs in via Google Workspace. Real credentials auth needs its own design pass (password storage, verification, reset, login rate-limiting) and is deliberately not half-built here.
  - `ApplyButton` reworked into per-artifact cards (Copy / PDF / Word each). `lib/api.ts` gains `apiFetchRaw` for binary passthrough (`apiFetch` always parses JSON and would corrupt a PDF).
  - **The `Fit Score: NN/100` contract (invariant #4) is untouched** — the split is a parser over existing output, so `FIT_RE`, `build_prompt` and their tests all stand.
  - Tests: +11 `test_e2e.py` (splitter incl. "evaluation never bleeds into exportable sections", no-header/empty/None/case-drift), +14 `test_api.py` (CV sync+async+shared poll+per-user isolation, PDF `%PDF` and DOCX zip magic, format/empty/413/auth rejection, filename-injection). Updated `used_this_month` to 4 — CVs count against the allowance, exports never do.
  - **Artifacts tab: not needed for this, and not built.** Downloads require zero persistence. A tab only earns its keep if drafts should survive a refresh and be revisited — a separate, weaker product question left open.

### Added — users table (email in a real table, not the twin JSON blob)
- **The email PII footgun is closed.** Pre-users-table releases stored the user's email in `career_twins.data.email` — a field any admin querying twins for prompt QA would incidentally see. New `users` table (id = Google `sub`, plus email/name/email_verified/notifications_enabled/created_at/last_seen_at) is now the source of truth. `db.upsert_user` is idempotent, keeps `notifications_enabled` and `created_at` across returning-user calls, and bumps `last_seen_at` on every hit. `db.get_user` returns booleans as Python bools. `db.set_notifications_enabled` flips the digest opt-in. `clear_user` extended — the users row is wiped alongside every other user-scoped table (NDPA/GDPR coverage).
- **`POST /user/session`** — the web tier's `(app)/layout.tsx` calls this on every authenticated render (adds one line to the existing `auth()` check). Body carries `email + name + email_verified` from the NextAuth session. Malformed email → 422 with a generic message (no PII in traces). **`PATCH /user/notifications`** — flips the digest toggle; 409 if the user has never established a session row (impossible from the web tier, but a direct API caller could hit it out of order).
- **Notify worker migrated to the new source of truth.** `LEFT JOIN users` in the digest query; email from `users.email` wins, falls back to `career_twins.data.email` for one release. Opt-out check runs BEFORE the email lookup, so a user with digests disabled is `skipped_opted_out` in the summary (never "missing email" — that would nudge Joy to fix a backfill that isn't the real issue). Matches for opted-out users stay `notified=0` so a future re-enable can catch them up.
- **Backfill path**: users with pre-migration twins hit `POST /user/session` on their next dashboard render → row created with current email → next digest run uses `users.email`.
- New offline checks: 13 in `test_db.py` (upsert idempotency, `last_seen_at` bumps, notifications toggle, empty-input rejection, clear_user coverage), 5 in `test_api.py` (session accept/reject, notifications 409, invalid email 422, deletion coverage), 4 in `test_notify_worker.py` (users.email wins, opted-out skip counter, opt-out preserves `notified=0`).

### Added — manual curation ATS handler for Nigerian & non-ATS employers

Missing `posted_at` is tolerated (treated as fresh); a bad date format is included rather than hidden (curator will notice on next update). Malformed entries (non-dict, missing keys) are skipped without killing the rest of the file — one bad row never destroys a company's entire ingest.
- **`_ATS_HANDLERS["manual"]` registered** — same dispatcher, same `(fetched, written)` return contract, same `_derive_company_domain` company-domain guess (so `verify_employers` can score Nigerian banks against their real domain, not the manual/ prefix). `_strip_html` + `_is_remote` heuristics apply so curated descriptions are cleaned and remote-mentioning locations flip correctly.

- 8 new offline checks in `test_ingest.py`: fresh + malformed row counts, stale skipping, `is_remote` heuristic, source_key namespacing, `company_domain` derivation, missing/broken file graceful handling, dry-run doesn't write, handler is registered in the dispatch dict.

## [0.5.0] — 2026-07-04 — Employer verification (scam protection)
### Added
- `verify.py`: deterministic trust engine. Score (0–100) computed in code from named evidence: free-mail vs corporate contact domain, DNS + MX records, live website, company-name/domain consistency, scam-pattern lexicon (fee requests, WhatsApp-only contact, crypto salaries, unrealistic pay). Any red flag caps the score at 35; no contact info caps at 40 ("unverified", never guessed).
- Gemini's only verification role: one narrow judgment on fetched homepage content (consistent / inconsistent / unclear); errors degrade to "unknown".
- Trust column (🟢🟡🟠🔴) in match results; automatic verification + low-trust warning on `apply`; `verify 2` / `verify info@company.com` shows full evidence. Per-session domain cache (network probed once per employer).
- `test_verify.py`: 27 offline checks with injected DNS/HTTP/LLM fakes covering every scoring path.
- New deps: `requests`, `dnspython`. Job extraction now also captures the contact field.
### Known limits (documented in README)
- No WHOIS/domain-age, no LinkedIn/Glassdoor checks; a scammer with a real website can still pass. Risk reduction, not a guarantee.

- Reviewer pass now enforces the style guide and re-checks over-generous fit scores.

### Changed — code-review fixes, employer onboarding redesign, admin credit grants (2026-07-17)
- **Code review (4 findings from `/code-review` on the merged PR #4):**
  - Client `splitDraft` (`web/lib/generate.ts`) only matched `## Fit Evaluation`, not the `## Fit Score` variant the Python side already handled — so on a `## Fit Score` draft the evaluation was routed into the CV-bullets bucket, mislabeling the candidate's gap analysis and leaving the "For your eyes only" panel empty. Regex now mirrors `core._EVAL_HEADER` (`fit (?:evaluation|score)`). This was the exact client/server divergence the prior fix claimed to eliminate — it survived because only the Python side got the shared header.
  - `split_application` / `splitDraft` silently dropped any preamble before the first header (the "never lose the draft" fallback only fired when *no* header matched). Pre-header text now folds into the cover letter — the exported artifact — on both sides. Regression test added.
  - Removed **55 tracked `.fuse_hidden*` FUSE-orphan files** across the repo and added a `.fuse_hidden*` gitignore rule (PR #4's `git mv` had propagated 5 of them into the employer route group; the rest predated it).
  - Dashboard no longer treats an unexpected error (500/401 — misconfig or a bug) identically to a transient outage: it still falls back to the sample dashboard (so a returning user isn't trapped), but `console.error`s the non-`ApiUnavailableError` case so a broken backend is diagnosable instead of silently masquerading as working demo data.
- **Employer onboarding redesigned.** The old form auto-wrote a name-guessed domain into the field on every keystroke ("I typed *supplya* and it chose supplya.com before I could edit it"). Now: the domain is **prefilled from the signed-in work-email domain** (a real signal — Google already verified control of that mailbox; free-mail like gmail is ignored), the name-based guess is a **clickable suggestion chip** the user opts into rather than an auto-fill, the signed-in identity is shown so setup isn't blind, and the copy is honest that the Verified badge comes later via domain confirmation. Email/name injected through the `(employer)` layout meta tags (same pattern as `create-career-twin`).
- **Admins can grant free unlock credits.** New `GET /admin/employers` (every employer + live ledger balance) and `POST /admin/employers/{id}/credits` (grant/claw back, `|delta| <= 500`, zero rejected, clawback can't drive the balance negative, recorded with an `admin:` reason prefix and an `AdminGrantedCredits` event; no `paystack_reference` so it never collides with the webhook's replay-dedup). Web proxy routes gated by `isAdmin`, and an "Employers & credits" section on `/admin` with a per-employer grant control. Credits are not verification — granting them never touches the trust badge. +10 API tests.

### Added — generic career-page connector (paste any job URL) + Jooble/Adzuna aggregators (2026-07-18)
- **Paste-a-URL now works for any job page, not just the 5 ATS JSON APIs.** The employer "Post a role" URL box only understood Greenhouse/Lever/Ashby/Workable/SmartRecruiter API hosts; a company careers page, a niche board (e.g. greenjobs.co.uk), or even a Greenhouse *embed* returned None → "we couldn't extract that URL." New **generic career-page connector**: when no ATS pattern matches (and the host isn't on the refuse-list), `core.fetch_url_text` fetches the page HTML, reduces it to readable text (`_html_to_text` — drops script/style/comments, keeps block structure, unescapes entities), and the existing `extract_single_job` Gemini path extracts the role. Endpoint reports `extracted_from: "url_generic"`.
  - **SSRF-hardened** because the server fetches user-supplied URLs: http(s) only; IP-literal hosts in private/loopback/link-local/reserved ranges (incl. `169.254.169.254` cloud metadata) refused up front with no DNS (so the guard holds even against an injected fetcher); hostnames that *resolve* to private IPs refused at fetch time via `getaddrinfo`; 2 MB read cap; non-HTML content-type rejected; pages under 200 chars of text treated as JS-shell/bot-wall → None. LinkedIn/Indeed/Workday/Taleo stay explicitly refused with guidance.
  - Tests: +14 in `test_ingest.py` (readable page → text, script/style/entity handling, too-short → None, **8 SSRF targets refused**), +2 in `test_employer.py` (non-ATS URL → `url_generic`; unreadable non-ATS URL → 422). Post-a-role copy updated to "a careers page, a job board, or an ATS".
- **Jooble and Adzuna aggregator sources** (the remaining agreed roadmap sources), env-gated so an unconfigured deploy stays silent. `workers/ingest_jobs.py` gains `_ingest_jooble` (POST via new `_post_json` seam, `JOOBLE_API_KEY`, token = `location:keywords`) and `_ingest_adzuna` (GET, `ADZUNA_APP_ID`+`ADZUNA_APP_KEY`, token = `country:keywords`, `ADZUNA_COUNTRY` default) with `normalize_jooble_job`/`normalize_adzuna_job` producing the shared job dict (HTML-stripped descriptions, `_is_remote` heuristic, Adzuna salary-range formatting). Registered in `_ATS_HANDLERS`; unknown/misconfigured rows skip gracefully like every other source. 4 `active:false` seed queries added to `data/job_sources.json` with a `_doc` note (set keys, tune queries, flip active via `PATCH /admin/job-sources/{id}/toggle` — no redeploy). Env keys registered in `render.yaml` (all `sync:false`). Tests: +8 in `test_ingest.py` (env-gated no-op, token parsing into the POST body / GET query, normalization, salary/remote handling, titleless-row skip).
